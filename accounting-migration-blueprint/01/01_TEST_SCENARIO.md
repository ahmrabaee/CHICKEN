# Blueprint 01 — سيناريو اختبار شامل
## Chart of Accounts Rebuild (إعادة بناء دليل الحسابات)

**الإصدار:** 1.0  
**التاريخ:** فبراير 2026  
**آخر تحديث:** يعكس تنفيذ العرض الشجري، فحص الحذف، وقيود اليومية بـ accountId

---

## الهدف

اختبار كامل للتعديلات المرتبطة بـ Blueprint 01 (إعادة بناء دليل الحسابات) من البداية، يشمل:
- **العرض الشجري** للحسابات (Tree View) مع توسيع/طي
- **إنشاء وتحديث وحذف** الحسابات
- **فحص إمكانية الحذف** (can-delete) قبل الحذف
- **القيود Journal Entries** باستخدام accountId والحسابات القابلة للقيد فقط
- **منع القيد** على حسابات المجموعة أو المعطلة أو المجمّدة
- **ميزان المراجعة** و**دفتر الحساب**
- **فلترة حسابات المخزون** في الفروع

## 3. اختبار العرض الشجري (Tree View)

### 3.1 هيكل الشجرة

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 3.1 | تحقق من عرض الحسابات | تُعرض الحسابات بشكل **شجري** (شجرة) وليس جدولاً عادياً |
| 3.2 | لاحظ التقسيمات الرئيسية | توجد جذور مثل: أصول، خصوم، حقوق ملكية، إيرادات، مصروفات |
| 3.3 | اضغط على أيقونة التوسيع (▶ أو ▼) بجانب حساب له أبناء | ينكشف المستوى التالي من الحسابات |
| 3.4 | اضغط مجدداً للطي | يختفي المستوى التالي |
| 3.5 | تحقق من عرض الكود والاسم | كل صف يظهر: **كود الحساب** (مثل 1000) و**اسم الحساب** |
| 3.6 | تحقق من النوع (rootType) | ألوان أو شارات تميز الأنواع: أصول، خصوم، حقوق ملكية، إيرادات، مصروفات |

### 3.2 عرض تفاصيل الحساب (Profile View)

| الخطوة | الإجراء | النتيجة المتوقعة |

| 3.2.2 | تحقق من المحتوى | كود، اسم، نوع، نوع التقرير، مجموعة/دفتر، الحالة، الحساب الأب إن وُجد |
| 3.2.3 | اضغط **تعديل** | يغلق التفاصيل ويفتح حوار التعديل |
| 3.2.4 | اضغط **حذف** | يغلق التفاصيل ويظهر حوار تأكيد الحذف (إن أمكن الحذف) |

## 4. اختبار إنشاء حساب جديد

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 4.1 | اضغط زر **إضافة حساب** في تبويب دليل الحسابات | يفتح حوار إضافة حساب جديد (RTL) |
| 4.2 | أدخل **كود الحساب:** `1199` | يُقبل الحقل |
| 4.3 | أدخل **اسم الحساب:** `حساب تجريبي للاختبار` | يُقبل الحقل |
| 4.4 | اختر **الأب (Parent):** حساب مجموعة مثل "النقد والبنوك" أو اترك فارغاً للجذر | يُقبل الحقل |
| 4.5 | اختر **نوع الحساب (accountType):** Bank أو Cash أو Receivable | يُقبل الحقل |
| 4.6 | حدد **هل مجموعة (isGroup):** لا | الحساب سيكون قابلاً للقيد (Ledger) |
| 4.7 | احفظ | رسالة نجاح، ويظهر الحساب الجديد في الشجرة |
| 4.8 | تحقق من موضع الحساب | يكون تحت الأب المحدد إذا اخترت أباً |

## 6. اختبار فحص الحذف (can-delete) وحذف الحساب

### 6.1 حذف حساب فارغ (بدون قيود ولا أبناء)

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 6.1 | من الشجرة، اضغط على أيقونة الحذف بجانب الحساب `1199` | يظهر حوار تأكيد |
| 6.2 | في الحوار، تحقق من الرسالة | إما "هل أنت متأكد من حذف هذا الحساب؟" أو رسالة توضح عدم وجود قيود |
| 6.3 | اضغط **تأكيد** أو **حذف** | يتم الحذف، ويختفي الحساب من الشجرة |
| 6.4 | تحقق | الحساب لم يعد يظهر |

### 6.2 محاولة حذف حساب عليه قيود

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 6.5 | اختر حساباً **له قيود** (مثل بنك أو ذمم مدينة مستخدم في قيود سابقة) | — |
| 6.6 | اضغط على زر الحذف | يظهر حوار أو رسالة |
| 6.7 | تحقق من الرسالة | **"لا يمكن حذف الحساب لوجود قيود عليه"** أو ما شابه |
| 6.8 | لا يتم الحذف | الحساب يبقى في الشجرة |

### 6.3 محاولة حذف حساب له أبناء

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 6.9 | اختر حساباً **مجموعة** له حسابات فرعية (مثل "النقد والبنوك") | — |
| 6.10 | اضغط على زر الحذف | يظهر حوار أو رسالة |
| 6.11 | تحقق من الرسالة | **"لا يمكن حذف الحساب لوجود حسابات فرعية"** أو "احذف الحسابات الفرعية أولاً" |
| 6.12 | لا يتم الحذف | الحساب يبقى |

**API للتحقق:**

```
GET /accounting/accounts/{id}/can-delete
Response: { "canDelete": false, "hasEntries": true } أو { "hasChildren": true }
```

## 7. اختبار القيود Journal Entries

### 7.1 عرض قائمة القيود

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 7.1 | في صفحة المحاسبة، انتقل إلى تبويب **قيود اليومية** | تُعرض قائمة القيود |
| 7.2 | تحقق من الأعمدة | رقم القيد، التاريخ، إجمالي المدين، إجمالي الدائن، الحالة (مسودة/مُرحّل) |
| 7.3 | اضغط على أي قيد لفتح التفاصيل | يظهر مربع تفاصيل القيد |

### 7.2 تفاصيل القيد

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 7.4 | في مربع التفاصيل، تحقق من ظهور الأسطر | كل سطر: الحساب، المدين، الدائن، الوصف |
| 7.5 | تحقق من عرض الحساب | يُعرض **كود الحساب** و**اسم الحساب** (من `account` أو `accountCode`/`accountName`) |
| 7.6 | تحقق من المدين والدائن | الأرقام تظهر بشكل صحيح (المبالغ بالشواكل أو القروش) |
| 7.7 | تحقق من التوازن | إجمالي المدين = إجمالي الدائن |

### 7.3 إنشاء قيد يدوي (إن وُجد النموذج)

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 7.8 | اضغط **إنشاء قيد** أو **قيد يدوي** | يفتح نموذج إنشاء قيد |
| 7.9 | أدخل الوصف: `قيد اختباري` | — |
| 7.10 | في السطر الأول: اختر **حساب قابلاً للقيد** (Ledger، وليس مجموعة) | قائمة الحسابات تعرض فقط الحسابات التي `isGroup = false` |
| 7.11 | أدخل المدين: `1000` (= ₪10.00) | — |
| 7.12 | في السطر الثاني: اختر حساباً آخر قابلاً للقيد، أدخل الدائن: `1000` | — |
| 7.13 | احفظ | نجاح، ويُعرض القيد في القائمة |

**ملاحظة:** القيود تستخدم الآن `accountId` في الـ API. الـ DTO:

```json
{
  "description": "قيد اختباري",
  "entryDate": "2026-02-13",
  "lines": [
    { "accountId": 10, "debit": 1000, "credit": 0 },
    { "accountId": 11, "debit": 0, "credit": 1000 }
  ]
}
```

### 7.4 ترحيل القيد (Post)

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 7.14 | افتح تفاصيل قيد بحالة **مسودة** | يظهر زر **ترحيل القيد** |
| 7.15 | اضغط **ترحيل القيد** | يتحول القيد إلى **مُرحّل** |
| 7.16 | تحقق من الحالة | الشارة تتغير من "مسودة" إلى "مُرحّل" |

## 8. اختبار منع القيد على حسابات غير مؤهلة

### 8.1 منع القيد على حساب مجموعة (Group)

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 8.1 | استخدم API لإنشاء قيد يستخدم **accountId** لحساب `isGroup = true` | — |
| 8.2 | مثال: حساب "النقد والبنوك" (1110) إذا كان مجموعة | — |
| 8.3 | أرسل الطلب | الـ API يرفض الطلب |
| 8.4 | تحقق من رسالة الخطأ | **"لا يمكن القيد على حسابات المجموعة"** أو `POSTING_TO_GROUP_ACCOUNT` |

**مثال API:**

```
POST /accounting/journal-entries
Body: {
  "description": "قيد خاطئ",
  "lines": [
    { "accountId": 4, "debit": 1000, "credit": 0 },
    { "accountId": 5, "debit": 0, "credit": 1000 }
  ]
}
```
(افترض أن 4 أو 5 هو حساب مجموعة — سيُرجع 400 مع رسالة مناسبة)

### 8.2 منع القيد على حساب معطّل

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 8.5 | عطّل حساباً (isActive: false) عبر API | — |
| 8.6 | حاول إنشاء قيد يستخدم هذا الحساب | الـ API يرفض |
| 8.7 | تحقق من رسالة الخطأ | **"لا يمكن القيد على حساب معطّل"** أو `POSTING_TO_DISABLED_ACCOUNT` |

### 8.3 منع القيد على حساب مجمد

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 8.8 | جمّد حساباً (freezeAccount: true) عبر API | — |
| 8.9 | حاول إنشاء قيد يستخدم هذا الحساب | الـ API يرفض |
| 8.10 | تحقق من رسالة الخطأ | **"لا يمكن القيد على حساب مجمد"** أو `POSTING_TO_FROZEN_ACCOUNT` |

## 12. اختبارات الـ API المباشرة

| # | الطلب | النتيجة المتوقعة |
|---|-------|------------------|
| 12.1 | `GET /accounting/accounts` | قائمة الحسابات الكاملة |

## 13. اختبار أخطاء القيد (Error Handling)

| الخطوة | الإجراء | النتيجة المتوقعة |
|--------|---------|------------------|
| 13.1 | قيد غير متوازن (مدين ≠ دائن) | رسالة: "Unbalanced entry" أو "يجب أن يتساوى المدين مع الدائن" |
| 13.2 | قيد بحساب غير موجود (accountId خاطئ) | خطأ 400 أو 404 |
| 13.3 | قيد بحساب مجموعة | خطأ `POSTING_TO_GROUP_ACCOUNT` |
| 13.4 | قيد بحساب معطّل | خطأ `POSTING_TO_DISABLED_ACCOUNT` |
| 13.5 | قيد بحساب مجمد | خطأ `POSTING_TO_FROZEN_ACCOUNT` |

## 14. ترتيب التنفيذ الموصى به (من الصفر)

```
1. تسجيل الدخول (admin / Admin@123)
2. المحاسبة → دليل الحسابات
3. التحقق من العرض الشجري وتوسيع/طي المستويات
4. إنشاء حساب جديد (1199)
5. تحديث الحساب (تغيير الاسم)
6. حذف الحساب 1199 (إن كان فارغاً)
7. قيود اليومية: عرض قائمة، فتح تفاصيل، التحقق من المدين/الدائن
8. (إن وُجد) إنشاء قيد يدوي بحسابات قابلة للقيد → ترحيل
9. ميزان المراجعة: التحقق من العرض والتوازن
10. اختبار منع الحذف: حساب عليه قيود، حساب له أبناء
11. اختبار منع القيد: حساب مجموعة عبر API
12. الفروع: التحقق من فلترة حسابات المخزون
```

## 16. استكشاف الأخطاء

| المشكلة | الحل |
|---------|------|
| الشجرة لا تظهر أو فارغة | تأكد من تشغيل `db:seed`؛ تحقق من `GET /accounting/accounts` |
| خطأ عند إنشاء قيد | تحقق من استخدام `accountId` وليس `accountCode`؛ تحقق أن الحساب `isGroup=false` |
| لا يمكن حذف الحساب | استدعِ `GET /accounts/:id/can-delete` لمعرفة السبب (قيود أو أبناء) |
| قائمة الحسابات في القيد تعرض مجموعات | استخدم `postableOnly=true` في `GET /accounting/accounts` |
| 401 عند API | أضف `Authorization: Bearer <token>` بعد تسجيل الدخول |
| مدين/دائن لا يظهر في تفاصيل القيد | الـ API يُرجع `debitAmount`/`creditAmount`؛ الواجهة تدعم كليهما |
| قاعدة بيانات قديمة | استخدم `npx prisma db push --force-reset` ثم `db:seed` (بيئة تطوير فقط) |

## 17. ملاحظات إضافية

- **الوحدات:** المبالغ في الـ API بصيغة **minor units** (قرش). ₪10.00 = 1000.
- **isGroup:** `true` = حساب تجميعي (لا يُسمح بالقيود)، `false` = دفتر (قابل للقيد).
- **rootType:** Asset, Liability, Equity, Income, Expense.
- **reportType:** Balance Sheet, Profit and Loss.
- **accountId vs accountCode:** القيود تستخدم `accountId`. الـ API يقبل `accountId` فقط في إنشاء القيود.
