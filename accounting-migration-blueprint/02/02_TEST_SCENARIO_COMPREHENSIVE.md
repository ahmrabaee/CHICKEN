## لمن هذا الشرح؟

## 1. ما هو محرك القيود العامة (GL Engine)؟

تخيل أن كل عملية في نظامك (بيع، شراء، دفعة، مصروف...) تنتج **قيوداً محاسبية**. قبل Blueprint 02:

- كل عملية كانت تنشئ القيود **بنفسها** بطريقة مختلفة
- إضافة مستند جديد يتطلب كتابة كود جديد في كل مرة
- لا يوجد **تسامح** للفروق الصغيرة (مثلاً 0.01 ₪ بسبب التقريب)
- القيود المتشابهة تُحفظ منفصلة بدلاً من دمجها

**محرك القيود العامة (GL Engine)** هو **نقطة دخول واحدة** لجميع أنواع القيود:

- يستقبل **خرائط القيود (GL Map)** من كل عملية
- يتحقق من التوازن مع **تسامح** للفروق الصغيرة
- يدمج القيود المتشابهة (نفس الحساب ومركز التكلفة)
- يضيف **قيد تدوير** تلقائياً عند الفروق الضئيلة
- يحفظ القيود بتنسيق موحد

---

## 2. ما كانت المشكلة قبل التحديث؟

| المشكلة | المعنى البسيط |
|---------|---------------|
| دمج منطق القيود مع العمليات | كل خدمة (SaleService، PurchaseService، إلخ) تنشئ القيود داخلها — تكرار ومنطق مشتت |
| عدم وجود GL Map | القيم تُنقل مباشرة للحفظ دون تحويل وسيط؛ لا يمكن تطبيق merge أو round-off |
| عدم وجود Round-Off Tolerance | فرق 0.01 يمنع الحفظ بدلاً من إنشاء قيد تدوير تلقائي |
| ضعف إعادة الاستخدام | نفس سلسلة التحقق (توازن، حفظ) مكررة في كل دالة |
| صعوبة إضافة مستندات جديدة | إضافة نوع مستند يتطلب دالة جديدة وتكرار منطق التحقق |

## 3. ماذا بنينا في Blueprint 02؟

### 3.1 GL Map — خريطة القيود الوسيطة

بدلاً من إنشاء قيد مباشرة، كل عملية تُرجع **قائمة GL Map**:

```
[
  { accountId: 1112, debit: 1000, description: 'Cash received' },
  { accountId: 1120, debit: 500, partyType: 'customer', partyId: 1 },
  { accountId: 4100, credit: 1500, description: 'Sales revenue' }
]
```

هذه القائمة تنتقل إلى **GL Engine** الذي يعالجها ويحفظها.

### 3.2 التسامح والتدوير (Tolerance & Round-Off)

| القاعدة | المعنى |
|---------|--------|
| فرق صغير ضمن tolerance | مثال: precision=2، tolerance=5 (0.05 ₪) — فرق 0.03 ₪ يُقبل ويُضاف قيد تدوير تلقائياً |
| فرق أكبر من tolerance | يُرفض الطلب مع رسالة UNBALANCED_ENTRY وعرض الفرق |
| حساب التدوير | من Company (مثلاً 5400 مصروفات أخرى) |

### 3.3 دمج القيود المتشابهة (Merge)

- نفس الحساب + نفس مركز التكلفة + نفس الطرف (إن وُجد) → **سطر واحد**
- تقليل حجم Ledger وحفظ أسهل

### 3.4 تطبيع القيم السالبة (Toggle Negative)

- إذا مدين ودائن سالبان ومتساويان: عكسهما لجعلهما موجبين
- إذا مدين سالب: نقل القيمة إلى دائن

### 3.5 عكس القيود (Reversal)

- عند إلغاء بيع أو دفعة: إنشاء **قيد عكسي** (مدين↔دائن) بدلاً من حذف القيد
- الحفاظ على Audit Trail

### 3.6 Feature Flag

- **gl_engine_enabled**: عند `true` تستخدم كل العمليات GL Engine؛ عند `false` تستخدم المسار القديم
- التحويل تدريجي وقابل للتحكم

## 4. ملخص في جملة واحدة

# القسم ك: ملاحظات تقنية للتسليم

## وحدات المبالغ

- المبالغ في الـ API بـ **minor units** (قرش): ₪10.00 = 1000
- في الواجهة: المستخدم يُدخل المبلغ **بالشيكل** (مثال: 10)؛ النظام يحوّله إلى 1000

## Tolerance و Precision

| الإعداد | المعنى | مثال |
|--------|--------|------|
| gl_debit_credit_tolerance=5 | 5 وحدات صغيرة = 0.05 ₪ عند precision=2 | الفروق ≤ 0.05 تُقبل |
| currency_precision=2 | منزلتان عشريتان | SAR, USD, EUR |
| threshold | 1/10^precision — أقل فرق يُضيف round-off | 0.01 ₪ |

## رموز الأخطاء

| الرمز | الرسالة العربية |
|-------|-----------------|
| UNBALANCED_ENTRY | القيد غير متوازن (مع عرض الفرق) |
| POSTING_TO_GROUP_ACCOUNT | لا يمكن القيد على حسابات المجموعة |
| POSTING_TO_DISABLED_ACCOUNT | لا يمكن القيد على حسابات معطلة |
| POSTING_TO_FROZEN_ACCOUNT | لا يمكن القيد على حسابات مجمدة |
| ALREADY_REVERSED | القيد معكوس بالفعل |
| ALREADY_POSTED | تم الترحيل مسبقاً |

## ترتيب تنفيذ موصى به (جميع الخطوات من الواجهة)

1. تسجيل الدخول (`admin` / `Admin@123`)
2. **الإعدادات** → تبويب **المحاسبة** — التحقق من GL Engine معطّل، اختيار **حساب فرق التدوير**
3. تفعيل GL Engine (Switch)
4. إنشاء **بيع** من المبيعات → التحقق من قيد البيع في **المحاسبة** → **قيود اليومية**
5. إنشاء **شراء** من المشتريات → التحقق من قيد الشراء
6. استلام/صرف **دفعة** من المدفوعات → التحقق من القيود
7. **إضافة قيد** يدوي متوازن (مدين 50، دائن 50) → نجاح
8. إضافة قيد غير متوازن (مدين 10، دائن 5) → التحقق من تعطيل الزر وعرض الفرق
9. **إضافة قيد** بفارق ضمن tolerance (مدين 10، دائن 10.03) → نجاح، التحقق من سطر "تدوير" في التفاصيل
10. فتح تفاصيل أي قيد → التحقق من مركز التكلفة، شارة التدوير
11. إلغاء بيع (Sale Void) → التحقق من قيد عكسي، **النقر على "عكس بواسطة #X"** لفتح القيد العكسي
12. تعطيل GL Engine → إنشاء قيد يدوي → التحقق من عمل المسار القديم
