# تحليل نقطة البيع (POS) — الأخطاء والعقبات وخطة الإصلاح

**التاريخ:** فبراير 2026  
**الهدف:** إكمال سيناريو اختبار Blueprint 03 (إنشاء زبون → فاتورة → دفعة → إلغاء)

---

## 1. ملخص سريع

نقطة البيع الحالية **واجهة وهمية (Mock)** ولا تتصل بالـ Backend. لا يمكن إنشاء فاتورة حقيقية منها، ولا يمكن اختيار زبون من القائمة. بدون إصلاحها، لا يمكن إكمال الاختبار.

---

## 2. الأخطاء والمعوقات المكتشفة

### 2.1 لا يوجد إنشاء فاتورة حقيقية

| الموقع | الوصف |
|--------|--------|
| `POS.tsx` سطر 333–337 | زر "إتمام البيع" يقوم فقط بـ `clearCart()` ولا يستدعي أي API |

```tsx
onClick={() => {
  // Save sale logic here
  clearCart();
}}
```

**النتيجة:** لا تُنشأ أي فاتورة في قاعدة البيانات.

---

### 2.2 لا يمكن اختيار زبون محدد

| الموقع | الوصف |
|--------|--------|
| `POS.tsx` سطور 176–195 | حقول نصية فقط: `customerName` و `customerPhone` |

- لا يوجد قائمة منسدلة لاختيار زبون من جدول `customers`
- لا يوجد ربط مع `customerId` للفاتورة
- الـ API يدعم `customerId` و `customerName` و `customerPhone`

**النتيجة:** لا يمكن ربط الفاتورة بزبون مسجل مسبقاً.

---

### 2.3 منتجات وهمية (Mock) وليست من المخزون

| الموقع | الوصف |
|--------|--------|
| `POS.tsx` سطور 24–32 | مصفوفة `products` ثابتة: IDs 1–6 وأسعار تقريبية |

```tsx
const products = [
  { id: 1, name: "فروج كامل", price: 22, ... },
  ...
];
```

**المشاكل:**
- IDs لا تطابق عناصر `items` الفعلية
- لا يوجد جلب من `useItems()` أو `itemService`
- الأسعار بالأرقام الكبيرة (₪22) بينما الـ API يتوقع `pricePerKg` بوحدات فرعية (مثلاً 2200 = 22.00)
- البذرة لا تنشئ items؛ العناصر تُنشأ يدوياً من واجهة المخزون

---

### 2.4 الكمية والوزن لا تتوافق مع الـ API

| الـ API يتوقع | POS الحالي |
|---------------|------------|
| `weightGrams` (جرام) | `quantity` (رقم صحيح) |
| `pricePerKg` (وحدات فرعية/كجم) | `price` (يبدو أنه بالريال) |

- المنتجات الوزنية تحتاج إدخال بالكيلو (مثلاً 2.5 كجم) وتحويلها إلى جرام
- المنتجات بالقطعة تحتاج منطق مختلف (عدد قطع بدل الوزن)

---

### 2.5 عدم وجود نوع البيع (Sale Type)

| المطلوب | الوضع الحالي |
|---------|--------------|
| `saleType: 'cash' \| 'credit' \| 'mixed'` | لا يوجد اختيار لنوع البيع |
| نقدي = مدفوع كاملاً | زر "نقداً" و"بطاقة" فقط لطريقة الدفع |
| آجل = إنشاء ذمة/دين | غير مدعوم |

للفواتير الآجلة (للمدفوعات اللاحقة) يجب اختيار `credit` أو `mixed`.

---

### 2.6 أزرار غير وظيفية

| الزر | السلوك الحالي |
|------|---------------|
| مسح (Barcode) | لا يفعل شيئاً |
| ميزان (Scale) | لا يفعل شيئاً |
| طباعة | لا يفعل شيئاً |

---

### 2.7 متطلبات المخزون للبيع

للبيع يجب:

1. وجود **Items** (من لوحة المخزون)
2. وجود **Inventory** و **InventoryLot** (عادةً عبر استلام مشتريات)

البذرة لا تنشئ items ولا مخزون. بدون مشترى مستلم، لا يمكن تنفيذ بيع فعلي.

---

## 3. خطة الإصلاح المقترحة

### 3.1 Frontend — POS.tsx

| المهمة | الوصف |
|--------|--------|
| **جلب المنتجات** | استخدام `useItems({ isActive: true })` وجلب العناصر النشطة |
| **اختيار الزبون** | إضافة `Select` أو Combobox لاختيار زبون من `useCustomers()`، مع خيار "زبون عادي" |
| **الكمية/الوزن** | إدخال بالكجم للأصناف الوزنية، وتحويل إلى `weightGrams` (× 1000) |
| **السعر** | استخدام `item.defaultSalePrice` (وحدات فرعية) وتحويل للعرض، والإرسال كما هو للـ API |
| **نوع البيع** | إضافة اختيار: نقدي / آجل / مختلط |
| **استدعاء API** | زر "إتمام البيع" يستدعي `useCreateSale()` ويرسل `CreateSaleDto` |
| **تحويل DTO** | تحويل السلة إلى `lines[]` بصيغة `{ itemId, weightGrams, pricePerKg, discountAmount? }` |

---

### 3.2 Backend — تعديلات اختيارية (للتجربة السريعة)

| المهمة | الوصف |
|--------|--------|
| **بذرة Items + مخزون** | إضافة دالة في `seed.ts` لإنشاء بضعة أصناف + مخزون أولي (مثلاً صنف اختبار مع lot ومخزون) |
| **دعم بيع بدون مخزون** | تفعيل `allowNegativeStock` لأصناف معينة أو إضافة وضع تجريبي للسماح بالبيع عند عدم وجود lots |

---

## 4. هيكل CreateSaleDto المطلوب

```ts
{
  customerId?: number,        // عند اختيار زبون
  customerName?: string,      // زبون عادي
  customerPhone?: string,
  saleType: 'cash' | 'credit' | 'mixed',
  discountAmount?: number,    // وحدات فرعية
  discountPct?: number,       // basis points (مثلاً 500 = 5%)
  lines: [
    {
      itemId: number,
      weightGrams: number,
      pricePerKg: number,      // وحدات فرعية
      discountAmount?: number
    }
  ],
  payments?: [                // اختياري عند البيع الآجل
    { amount: number, paymentMethod: 'cash'|'card'|... }
  ]
}
```

## 5. ترتيب التنفيذ المقترح

### المرحلة 1 — إصلاح POS الأساسي (Frontend)

1. ربط المنتجات بـ `useItems()`
2. ربط الزبائن بـ `useCustomers()` مع إمكانية "زبون عادي"
3. إضافة `saleType` (نقدي/آجل/مختلط)
4. استدعاء `salesService.createSale()` من زر "إتمام البيع"
5. عرض رسائل نجاح/خطأ والانتقال لقائمة المبيعات بعد النجاح

### المرحلة 2 — تجهيز البيئة (Backend + seed)

1. إضافة بذرة أصناف (مثلاً 3–5 أصناف) مع إنشاء `inventory` و `inventory_lots` بمخزون أولي
2. أو: شرح واضح لإعداد المشتريات أولاً ثم استلامها لإنشاء المخزون

### المرحلة 3 — تحسينات إضافية

1. ربط طباعة الإيصال بـ `getReceipt`
2. (اختياري) دمج الميزان والباركود عند توفر الأجهزة

## 6. مسار بديل للتجربة السريعة

إذا أُريد اختبار Blueprint 03 بدون انتظار اكتمال POS:

- **إنشاء الفاتورة من واجهة أخرى:** إضافة صفحة أو نموذج لإنشاء فاتورة بيع يدوياً (مثلاً تحت `/sales/create`) بصيغة أبسط: اختيار زبون، صنف، كمية، سعر، نوع البيع، وحفظ الفاتورة.

هذا يسمح باختبار: فاتورة → دفعة → إلغاء الدفعة، دون الاعتماد على نقطة البيع الحالية.

---

## 8. الخلاصة

نقطة البيع الحالية غير جاهزة للاستخدام الفعلي. الإصلاح يتطلب:

1. ربط المنتجات بالمخزون الفعلي
2. إضافة اختيار الزبون من قائمة العملاء
3. استدعاء إنشاء الفاتورة من الـ API عند "إتمام البيع"
4. تهيئة بيانات مخزون أولية (بذرة أو استلام مشتريات) لتمكين البيع

بدون هذه التعديلات، لا يمكن إكمال سيناريو اختبار Blueprint 03 من واجهة نقطة البيع الحالية.
