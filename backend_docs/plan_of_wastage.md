> **الملف:** `backend_docs/plan_of_wastage.md`  
> **تاريخ:** 2025-02-21  
> **الحالة:** خطة تصميمية شاملة

---

## 1. نموذج المخزون في النظام

### 1.1 الفكرة الأساسية

- **المخزون الخام** يُشترى من التجار كـ **فئة** وليس أصنافاً جاهزة.
- في نظامنا حالياً: فئة أساسية واحدة هي **"دجاج"**.
- هذا الدجاج يُقسّم إلى المنتجات التي نبيعها:

  | المنتج         | مثال       |
  |----------------|------------|
  | أجنحة دجاج     | Chicken Wings |
  | دجاج كامل      | Whole Chicken |
  | صدور دجاج      | Chicken Breast |
  | أفخاذ          | Chicken Thighs |
  | إلخ...         | ... |

### 1.2 لا شراء أصناف جاهزة

- **لا نشتري** من المستورد صنفاً معيناً (مثل "10 كيلو صدور").
- **نشتري** فئة (دجاج) بالكيلو، ثم نحوّل المخزون إلى أصناف داخل النظام.

---

## 2. صفحة تحويل المخزون (Stock Transfer)

### 2.1 الهدف

صفحة واحدة تسمح بتحويل المخزون الخام المشترى (دجاج) إلى المنتجات القابلة للبيع، مع:

1. ربط محاسبي واضح بكل الحركات.
2. تحديد تاريخ صلاحية لكل شحنة عند التحويل.
3. إمكانية **رد المخزون للمورد** عند التلف أو عدم الجودة (حالة نادرة لكن يجب تغطيتها محاسبياً).

### 2.2 سيناريو العمل

```
[شراء دجاج من تاجر]
        ↓
   مخزون خام (دجاج)
        ↓
[تحويل المخزون]
   - اختيار الشحنة/الكمية
   - تحديد تاريخ صلاحية
   - توزيع على المنتجات (أجنحة، صدور، إلخ)
        ↓
   مخزون منتجات (lots لكل منتج)
        ↓
[البيع] أو [الهدر] أو [الرد للمورد]
```

## 3. الفصل الهندسي بين المخزون القديم والجديد (حسب الصلاحية)

### 3.1 المشكلة

- مخزون قديم لمنتج ما لم يُنفّق كاملاً.
- وصول شحنة جديدة.
- للمنتج الواحد: كمية قديمة قريبة من انتهاء الصلاحية + كمية طازجة جديدة.
- المطلوب: فصل واضح بينهما بدون تعقيد.

### 3.2 الحل: نظام الـ Lots (الدُفعات)

كل **شحنة** عند تحويلها إلى منتجات تُسجّل كـ **Lot** منفصل لكل منتج:

| المنتج      | Lot  | الكمية | تاريخ الصلاحية | التكلفة |
|-------------|------|--------|----------------|---------|
| صدور دجاج   | L001 | 15 kg  | 2025-02-26     | 42 ₪/kg |
| صدور دجاج   | L002 | 8 kg   | 2025-02-22     | 40 ₪/kg |

- **L001** = شحنة جديدة (صلاحية أبعد).
- **L002** = شحنة قديمة (صلاحية أقرب).

### 3.3 منطق FIFO للبيع

- عند البيع يُخصَّص المخزون من الـ lots حسب **الأقدم من حيث الصلاحية** (أو `receivedAt`).
- البيع يأخذ أولاً من الـ lots القريبة من الانتهاء ثم الأحدث.
- بذلك نضمن: **لا خلط** بين الكميات القديمة والجديدة، والفصل يتم عبر الـ Lot نفسه.

### 3.4 بنية الجداول الداعمة (موجودة أو مطلوبة)

- **InventoryLot** (موجود): `itemId`, `expiryDate`, `remainingQuantityGrams`, `unitPurchasePrice`, `receivedAt`
- كل عملية تحويل مخزون تنشئ lots جديدة للمنتجات المستهدفة مع تواريخ صلاحية محددة.

## 4. صفحة تحويل المخزون — التصميم التفصيلي

### 4.1 الواجهة الرئيسية

```
┌─────────────────────────────────────────────────────────────────┐
│ تحويل المخزون                                                    │
├─────────────────────────────────────────────────────────────────┤
│ الشحنة المصدر:  [قائمة الشحنات المتاحة للتحويل]                   │
│   - رقم الشحنة | التاجر | الوزن الكلي | تاريخ الاستلام             │
│                                                                  │
│ تاريخ الصلاحية للشحنة: [تاريخ] (أيام افتراضية من الفئة)           │
│                                                                  │
│ التوزيع على المنتجات:                                            │
│   المنتج          │ الكمية (kg) │ التكلفة الموزعة │               │
│   أجنحة دجاج      │    5.0     │    150 ₪        │               │
│   صدور دجاج       │   10.0     │    350 ₪        │               │
│   دجاج كامل       │    3.0     │     66 ₪        │               │
│   ────────────────┼─────────────┼─────────────────│               │
│   المجموع         │   18.0     │    566 ₪        │               │
│                                                                  │
│ التحقق: المجموع ≤ وزن الشحنة المتاح                              │
│                                                                  │
│ [حفظ]  [إلغاء]                                                   │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 قواعد التحويل

1. **الكميات**: مجموع الكميات الموزعة لا يتجاوز وزن الشحنة المتاح.
2. **التكلفة**: تُوزَّع تناسباً أو حسب سعر افتراضي للمنتج، مع إمكانية التعديل اليدوي ضمن حدود المعقول.
3. **التاريخ**: تاريخ الصلاحية يُحدَّد عند التحويل؛ الافتراضي من `Category.defaultShelfLifeDays` أو `Item.shelfLifeDays`.

### 4.3 الحسابات المحاسبية عند التحويل

| الحركة          | المدين             | الدائن           |
|-----------------|--------------------|-------------------|
| خروج من مخزون الخام (دجاج) | -                  | حساب مخزون الخام |
| دخول لمخزون المنتجات      | حساب مخزون المنتجات | -                 |

- إذا كان المخزون الخام والمخزون المنتجات في نفس الحساب (مثل 1130): القيد يكون صافيًا داخليًا أو يُسجّل كتحويل داخلي.
- إذا اختلفت الحسابات: مدين مخزون المنتجات، دائن مخزون الخام.

### 4.4 Stock Ledger Entry

- `voucherType = 'stock_transfer'`
- `voucherId` = معرف سند التحويل
- لكل منتج: SLE موجب (دخول) للمنتج، SLE سالب (خروج) للمخزون الخام.

## 5. رد المخزون للمورد (Purchase Return)

### 5.1 متى تُستخدم

- المخزون لم يعجب (جودة، أو نوع غير مطابق).
- المخزون تالف.
- حالة **نادرة** لكن يجب أن تكون مغطاة في النظام.

### 5.2 التدفق

1. إنشاء **سند رد مشتريات** (Purchase Return) مرتبط بأمر الشراء الأصلي.
2. تحديد الكمية والمنتج/الفئة المرتجعة.
3. خصم من المخزون (InventoryLot / Inventory).
4. ترحيل محاسبي:

| القيد                    | المدين              | الدائن             |
|--------------------------|---------------------|---------------------|
| خصم المخزون             | حساب المورد (ذمم دائنة) | حساب المخزون       |
| أو: استرداد نقدي        | حساب المورد         | الصندوق/البنك      |

- في حال الائتمان: تقليل ذمم المورد، خصم المخزون.
- في حال الاسترداد النقدي: تقليل الذمم، خصم الصندوق.

### 5.3 التأثير على Stock Ledger

- SLE سالب (خروج) لحساب المخزون.
- `voucherType = 'purchase_return'`
- ربط بـ `purchaseId` الأصلي.

## 6. إدارة الصلاحية والإشعارات

### 6.1 القاعدة

- لكل **Lot** نحدّد `expiryDate` عند التحويل (أو الاستلام في حالة الشراء المباشر).
- النظام يتتبع الـ lots حسب صلاحيتها.

### 6.2 الإشعار عند قرب انتهاء الصلاحية

- عند اقتراب تاريخ الصلاحية (مثلاً: يوم واحد أو قابل للتكوين):
  - نظام خلفي (Cron/Job) يفحص lots حيث `expiryDate - now <= يوم واحد`.
  - إنشاء إشعار للمستخدم:

    > "الشحنة [رقم] - منتج [اسم] - تم شراؤها في [تاريخ]، تم تحويلها في [تاريخ]، وهي قريبة من التلف. هل تريد نقلها إلى سلة الهدر أم الاستمرار في البيع؟"

### 6.3 إجراء المستخدم

| الخيار          | الإجراء                                              |
|-----------------|------------------------------------------------------|
| نقل للهدر       | إنشاء WastageRecord، خصم من Lot، قيد محاسبي للهدر   |
| الاستمرار       | إبقاء Lot كما هو، إمكانية إغلاق الإشعار دون تغيير    |

## 7. حركة الهدر والمحاسبة عليها

### 7.1 أنواع الهدر (موجودة في النظام)

- `spoilage` — تلف
- `trimming` — قص وتشذيب
- `expired` — منتهي الصلاحية
- `damaged` — تلف مادي
- `end_of_day` — هدر نهاية اليوم
- `other` — أخرى

### 7.2 الترحيل المحاسبي للهدر

| القيد        | المدين                   | الدائن         |
|-------------|---------------------------|----------------|
| مصروف الهدر | 5300 مصروف الهدر (Wastage) | حساب المخزون  |

- القيمة من `WastageRecord.estimatedCostValue` أو من تكلفة الـ Lot المعني (FIFO).
- SLE: `voucherType = 'wastage'`, `voucherId` = معرف سجل الهدر.

### 7.3 ربط الهدر بالـ Lot

- `WastageRecord.lotId` يربط بالسجل المعين.
- عند التسجيل: خصم من `InventoryLot.remainingQuantityGrams` وإنشاء SLE سالب للمخزون.

## 8. التعديلات المطلوبة على نموذج البيانات

### 8.1 نموذج الشراء الحالي

- **Purchase** و **PurchaseLine** مرتبطان بـ **Item** (منتج).
- للتطابق مع نموذج "الشراء بالفئة":
  - **الخيار أ:** إضافة فئة شراء مثل "دجاج" وتوسيع Purchase ليدعم الشراء بالفئة مع سطر واحد أو أكثر للوزن الإجمالي.
  - **الخيار ب:** الإبقاء على Purchase كما هو، وإنشاء **Category Stock** (مخزون الفئة) يُستقبل فيه الدجاج الخام، ثم يُحوَّل عبر سند التحويل.

### 8.2 جداول مقترحة لتحويل المخزون

```
StockTransfer (سند التحويل)
├── id
├── transferNumber (رقم فريد)
├── sourceType ('category_purchase' | 'purchase') 
├── sourceId (معرف المصدر: purchaseId أو categoryStockId)
├── transferDate
├── branchId
├── expiryDate (صلاحية الشحنة بعد التحويل)
├── totalWeightGrams
├── totalCostValue
├── status ('draft' | 'completed' | 'cancelled')
├── createdById
├── completedAt
└── notes

StockTransferLine (تفاصيل التحويل)
├── id
├── stockTransferId
├── itemId (المنتج المستهدف)
├── weightGrams
├── unitCost (تكلفة الكيلو الموزعة)
├── lineCostValue
├── lineNumber
└── [عند الحفظ: إنشاء InventoryLot للمنتج]
```

### 8.3 مخزون الفئة (Category Stock) — إن لزم

```
CategoryStock (مخزون الخام حسب الفئة)
├── id
├── categoryId
├── branchId
├── purchaseId (أو مرجع الشحنة)
├── totalWeightGrams
├── remainingWeightGrams
├── unitCost
├── receivedAt
├── expiryDate (اختياري للخام قبل التحويل)
└── lotNumber
```

- أو: استخدام **Item** خاص باسم "دجاج خام" ضمن الفئة، ومعاملته كمنتج وسيط قبل التحويل إلى المنتجات النهائية.

## 10. خطة التنفيذ المقترحة (مراحل)

### المرحلة 1 — تحويل المخزون الأساسي

1. إضافة جدول **StockTransfer** و **StockTransferLine**.
2. إنشاء API لتحويل المخزون:
   - إدخال المصدر (شحنة/شراء)، تاريخ الصلاحية، التوزيع على المنتجات.
   - التحقق من عدم تجاوز الكميات.
3. عند التأكيد: إنشاء **InventoryLot** لكل منتج مع `expiryDate` المحدد.
4. تحديث **Inventory** و **StockMovement** و **StockLedgerEntry**.

### المرحلة 2 — المحاسبة

1. ربط تحويل المخزون بالقيود (تحويل داخلي أو بين حسابات مخزون).
2. تطبيق منطق رد المشتريات مع القيود المناسبة.
3. التأكد من ترحيل الهدر (5300) بشكل صحيح.

### المرحلة 3 — الصلاحية والإشعارات

1. إضافة Job يفحص الـ lots القريبة من انتهاء الصلاحية.
2. إنشاء إشعارات للمستخدم مع خيار "نقل للهدر" أو "متابعة البيع".
3. ربط زر "نقل للهدر" بإنشاء `WastageRecord` والقيد المحاسبي.

### المرحلة 4 — رد المخزون للمورد

1. إنشاء نموذج **Purchase Return**.
2. ربطه بأمر الشراء الأصلي.
3. تطبيق القيود المحاسبية و Stock Ledger.

## 12. الفصل بين المخزون القديم والجديد — الخلاصة

| العنصر             | الآلية                                             |
|--------------------|----------------------------------------------------|
| الفصل              | كل Lot = دفعة واحدة مع `expiryDate` وكمية و تكلفة |
| البيع (FIFO)       | الخصم من أقدم الـ lots (حسب الصلاحية أو الاستلام) |
| الإشعار            | عند اقتراب انتهاء صلاحية Lot                      |
| الهدر              | خصم من Lot المحدد، قيد مصروف الهدر، SLE سالب      |

بهذا يتحقق فصل واضح وبسيط بين المخزون القديم والجديد دون خلط في الكميات أو التواريخ.
