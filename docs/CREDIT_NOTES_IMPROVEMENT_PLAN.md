# خطة تحسين الإشعارات الدائنة (Credit Notes)

## 1. نظرة عامة

تحويل صفحة الإشعارات الدائنة من نموذج بسيط داخل popup إلى **صفحة profile كاملة** مع تدفق اختيار ذكي: الطرف أولاً → الفواتير المرتبطة به → المبلغ والسبب. رقم الفاتورة يُختار تلقائياً من القائمة (لا إدخال يدوي لـ ID).

---

## 2. الهيكل المقترح

### 2.1 المسارات (Routes)

| المسار | الوظيفة |
|-------|----------|
| `/credit-notes` | قائمة الإشعارات + زر "إنشاء إشعار دائن" |
| `/credit-notes/new` | صفحة إنشاء إشعار دائن (Profile كامل) |
| `/credit-notes/:id` | عرض تفاصيل إشعار دائن محدد (Profile للعرض/الترحيل) |

### 2.2 تدفق المستخدم (User Flow)

```
1. المستخدم يذهب إلى /credit-notes/new
2. يختار نوع العملية: فاتورة بيع ← عميل | فاتورة شراء ← مورد
3. يبحث ويختار الطرف (عميل أو مورد) من Combobox قابل للبحث
4. تظهر تلقائياً الفواتير المفتوحة لهذا الطرف (outstanding > 0) مرتبة بالتاريخ
5. يختار الفاتورة من القائمة (اختيار تلقائي - لا إدخال ID)
6. يظهر المستحق الحالي للفاتورة المختارة
7. يدخل المبلغ (مع تحقق: لا يتجاوز المستحق)
8. يدخل السبب (اختياري)
9. ينقر "إنشاء" → يُنشأ الإشعار بحالة مسودة
10. يُعاد توجيهه إلى /credit-notes/:id أو يبقى في الصفحة مع خيار "ترحيل" أو "إنشاء آخر"
```

---

## 3. مكونات الواجهة (Frontend)

### 3.1 صفحة القائمة `/credit-notes` (CreditNotes.tsx)

- **الاحتفاظ بـ**: جدول الإشعارات، زر "إنشاء إشعار دائن"
- **التعديل**: زر "إنشاء" ينقل إلى `/credit-notes/new` بدلاً من فتح Dialog
- **إضافة**: عمود "الطرف" (اسم العميل/المورد) في الجدول إن أمكن
- **إضافة**: رابط من كل صف إلى `/credit-notes/:id` لعرض التفاصيل

### 3.2 صفحة الإنشاء `/credit-notes/new` (CreditNoteProfile.tsx - وضع إنشاء)

**Layout**: صفحة كاملة (مثل PaymentProfile، ExpenseProfile) مع:

1. **رأس الصفحة**
   - عنوان: "إنشاء إشعار دائن"
   - زر رجوع إلى `/credit-notes`
   - زر "إنشاء" (أو "حفظ كمسودة")

2. **البطاقة الأولى: اختيار الطرف**
   - Tabs أو Select: "فاتورة بيع" | "فاتورة شراء"
   - **Party Combobox** (قابل للبحث):
     - للبيع: `CustomerSearchCombobox` أو مكون مشابه يعرض العملاء
     - للشراء: `SupplierSearchCombobox` يعرض الموردين
   - البحث بالاسم، الهاتف (للعملاء)، أو أي معرّف متاح
   - عرض: الاسم، الهاتف (إن وجد)، وربما المستحق الإجمالي

3. **البطاقة الثانية: اختيار الفاتورة** (تظهر بعد اختيار الطرف)
   - جدول أو قائمة بالفواتير المفتوحة (outstanding > 0)
   - مصدر البيانات: `reconciliationService.getOpenInvoices(partyType, partyId)`
   - الأعمدة المقترحة:
     | رقم الفاتورة | التاريخ | المبلغ الإجمالي | المستحق | إجراء |
   - ترتيب: حسب التاريخ (الأقدم أولاً أو الأحدث - حسب تفضيل المحاسبة)
   - **اختيار الفاتورة**: Radio أو Click على الصف (لا إدخال ID يدوي)
   - عند الاختيار: عرض المستحق بوضوح وتفعيل حقل المبلغ

4. **البطاقة الثالثة: تفاصيل الإشعار**
   - **المبلغ (شيكل)** *: NumericInput
     - التحقق: `amount <= outstanding`
     - عرض: "الحد الأقصى: ₪ X.XX"
   - **السبب (اختياري)**: Input أو Textarea
   - **ملخص**: الفاتورة المختارة، الطرف، المبلغ، السبب

### 3.3 صفحة عرض الإشعار `/credit-notes/:id` (CreditNoteProfile.tsx - وضع عرض)

- عرض تفاصيل الإشعار: الرقم، التاريخ، الفاتورة الأصلية، الطرف، المبلغ، الحالة، السبب
- زر "ترحيل" إذا كانت الحالة مسودة
- زر "إنشاء إشعار آخر" ينقل إلى `/credit-notes/new`
- روابط سريعة: إلى الفاتورة الأصلية (Sale/Purchase)، إلى الطرف (Customer/Supplier)

## 4. المكونات والخدمات المطلوبة

### 4.1 مكونات جديدة أو معدّلة

| المكون | الوصف |
|--------|-------|
| `PartySearchCombobox` | Combobox عام: يدعم العملاء أو الموردين حسب النوع، مع بحث بالاسم/الهاتف |
| أو إعادة استخدام `CustomerSearchCombobox` + إنشاء `SupplierSearchCombobox` | للاتساق مع POS والدفعات |
| `OpenInvoiceTable` | جدول يعرض الفواتير المفتوحة مع إمكانية الاختيار (مشابه لـ Reconciliation) |
| `CreditNoteProfile` | صفحة موحدة: new = إنشاء، :id = عرض/ترحيل |

### 4.2 الخدمات (APIs) الموجودة

- `reconciliationService.getOpenInvoices(partyType, partyId)` ✅
- `reconciliationService.getSaleOutstanding(saleId)` / `getPurchaseOutstanding(purchaseId)` ✅
- `customerService.getCustomers({ search })` / `searchCustomers(q)` ✅
- `supplierService.getSuppliers({ search })` ✅
- `creditNoteService.create(dto)` / `submit(id)` ✅

### 4.3 تحسينات Backend (اختيارية)

- **إضافة `search` لـ Suppliers**: إذا لم يكن موجوداً، إضافة `?search=` في `getSuppliers` أو إنشاء `/suppliers/search?q=`
- **ترتيب open-invoices**: التأكد من ترتيب الفواتير حسب `postingDate` (الأحدث أولاً أو الأقدم - حسب السياسة)

---

## 5. رقم الفاتورة (ID) — تلقائي

- **لا يوجد حقل "رقم الفاتورة (ID)"** في النموذج
- المستخدم يختار الفاتورة من القائمة المنسدلة/الجدول
- القيمة `originalInvoiceId` تُستمد تلقائياً من الفاتورة المختارة
- `originalInvoiceType` يُستمد من نوع الطرف المختار (عميل = sale، مورد = purchase)

## 6. اقتراحات إضافية

### 6.1 تجربة المستخدم (UX)

1. **فلترة العملاء/الموردين**: عرض فقط من لديهم فواتير مفتوحة (outstanding > 0) — يتطلب API جديد أو فلترة على الواجهة بعد جلب القائمة
2. **اقتراح الفاتورة**: إذا كان للعميل/المورد فاتورة واحدة مفتوحة، اختيارها تلقائياً
3. **تعبئة المبلغ**: زر "المبلغ الكامل" لملء المبلغ بالمستحق بالكامل
4. **تحذير قبل الترحيل**: تأكيد قبل ترحيل الإشعار (مثل DocumentStatusBadge / CancelConfirmDialog)
5. **Breadcrumb**: الإشعارات الدائنة > إنشاء جديد | الإشعار #CN-000001

### 6.2 إمكانية الوصول والوضوح

1. **عرض اسم الطرف** في قائمة الإشعارات (عمود إضافي)
2. **رابط سريع** من الإشعار إلى الفاتورة الأصلية وإلى الطرف
3. **حالة التحميل**: Skeleton أو Spinner واضح عند جلب الفواتير
4. **رسائل خطأ واضحة**: "المبلغ يتجاوز المستحق"، "اختر الطرف أولاً"، "اختر الفاتورة"

### 6.3 تحسينات تقنية

1. **تفعيل React Query** لـ `open-invoices` مع `enabled: !!partyId`
2. **Debounce** على بحث الطرف لتحسين الأداء
3. **التحقق من الصلاحيات**: التأكد من أن الصفحة محمية بـ Roles (admin, accountant)

### 6.4 تحسينات محاسبية (اختيارية)

1. **إضافة partyType/partyId** لقيود الإشعار الدائن في GL لتحسين التقارير
2. **تاريخ الإشعار**: السماح بتعديل تاريخ الإشعار عند الإنشاء (حالياً يُستخدم `new Date()`)

## 8. ملخص الفرق بين الحالي والمقترح

| العنصر | الحالي | المقترح |
|--------|--------|---------|
| واجهة الإنشاء | Popup (Dialog) | صفحة Profile كاملة |
| اختيار الطرف | غير موجود | Combobox قابل للبحث (عميل/مورد) |
| اختيار الفاتورة | إدخال ID يدوي | اختيار من قائمة الفواتير المفتوحة مرتبة بالتاريخ |
| رقم الفاتورة | يدوي | تلقائي من الاختيار |
| عرض الإشعار | ضمن الجدول فقط | صفحة Profile منفصلة |
| الترحيل | من الجدول | من صفحة Profile أو الجدول |

- `Reconciliation.tsx` — تدفق اختيار الطرف ثم الفواتير
- `PaymentProfile.tsx` / `ExpenseProfile.tsx` — هيكل صفحة Profile
