# خطة الدفع المختلط — نقدي + آجل (Partial Payment: Cash + Deferred Debt)

## الوضع الحالي

الـ POS يدعم حالياً نمطين فقط عبر مفتاح `saleType`:
| النمط | السلوك |
|---|---|
| `cash` | يدخل الكاشير مبلغاً؛ إذا كان أقل من الإجمالي يُرحَّل الباقي كدين تلقائياً (لكن الواجهة تعرضه كخطأ/تحذير فقط) |
| `credit` | المبلغ الكامل دين على الزبون |

**المشكلة:** لا توجد واجهة واضحة تُظهر للكاشير صراحةً "المبلغ النقدي = X" و"المبلغ الآجل = Y"، وهو ما يطلبه المستخدم.

---

## الهدف

إضافة **نمط دفع مختلط (Mixed)** حيث يُحدِّد الكاشير يدوياً:
- المبلغ المدفوع **نقداً** (أو بطاقة)
- الباقي يُرحَّل تلقائياً إلى **حساب الزبون كآجل (دين)**

---

## 1. تغييرات الـ Frontend — `app/frontend/src/pages/POS.tsx`

### 1.1 إضافة نمط `"mixed"` إلى `saleType`

```typescript
// تغيير نوع الحالة
const [saleType, setSaleType] = useState<"cash" | "credit" | "mixed">("cash");
```

### 1.2 أزرار اختيار نمط الدفع

استبدال الزرين (نقدي / آجل) بثلاثة أزرار:

```
┌──────────┬──────────┬─────────────────────────────┐
│  نقداً   │   آجل   │  جزئي (نقدي + آجل)          │
└──────────┴──────────┴─────────────────────────────┘
```

### 1.3 واجهة نمط "مختلط"

عند اختيار "جزئي (نقدي + آجل)" تظهر:

```
┌─────────────────────────────────────────────────────┐
│  الإجمالي: ₪80.00                                   │
│                                                     │
│  المبلغ النقدي:  [________] ₪  (يُدخله الكاشير)    │
│  المبلغ الآجل:   ₪XX.XX    (يُحسَب تلقائياً)        │
│                                                     │
│  طريقة الدفع النقدي: [نقداً] [بطاقة]               │
└─────────────────────────────────────────────────────┘
```

**منطق الحساب:**
```typescript
const cashAmountMinor  = toMinor(paidAmount);           // ما أدخله الكاشير
const deferredMinor    = Math.max(0, totalMinor - cashAmountMinor); // الباقي = آجل
```

**التحقق:**
- `cashAmountMinor > 0` (يجب أن يكون هناك مبلغ نقدي فعلاً)
- `cashAmountMinor < totalMinor` (وإلا لماذا لا يستخدم "نقدي" فقط؟)
- بيانات الزبون إلزامية دائماً في هذا النمط (يجب لتسجيل الدين)

### 1.4 منطق `handleCompleteSale`

```typescript
if (saleType === "mixed") {
  const cashAmount = Math.min(Math.max(0, cashAmountMinor), totalMinor);
  dto.saleType = "cash";          // يُرسَل للباكند كـ cash مع دفع جزئي
  dto.payments = [
    {
      amount: cashAmount,
      paymentMethod: paymentMethod === "cash" ? "cash" : "card",
    },
  ];
  // الباقي (totalMinor - cashAmount) يُرحَّل تلقائياً كدين في الباكند
}
```

> **ملاحظة:** الباكند الحالي يدعم هذا السيناريو بالكامل — عند إرسال `payments` بمبلغ أقل من الإجمالي يُنشئ الباكند تلقائياً سجل دين للباقي.

### 1.5 عرض الفاتورة والإيصال

يجب أن يعرض الإيصال الحراري:
```
المدفوع نقداً:  ₪50.00
المرحَّل آجلاً: ₪30.00
─────────────────────
الإجمالي:       ₪80.00
```

تعديل في `ThermalReceipt` لعرض سطرين بدل سطر واحد عندما `deferredAmount > 0`.

## 2. تغييرات الـ Backend — لا تغييرات جوهرية مطلوبة ✅

الباكند الحالي يُعالج هذا تلقائياً:

```typescript
// في sales.service.ts
const remaining = totalAmount - totalPayments;  // الفرق = آجل

// إذا remaining > 0 يُنشئ سجل debt ويُحدِّث currentBalance للزبون
```

### 2.1 التحقق من الـ DTO (اختياري)

يمكن إضافة `saleType: "mixed"` للـ DTO للوضوح، أو الإبقاء على `"cash"` مع دفع جزئي.

**التوصية:** الإبقاء على الـ DTO الحالي `saleType: 'cash' | 'credit'` — نمط "mixed" هو فقط مفهوم واجهة المستخدم، والباكند يُعالجه كـ `cash` بدفع جزئي.

### 2.2 قيود الباكند الموجودة (تبقى كما هي)

```typescript
// partial payment: customer info required
if (remaining > 1 && !hasCustomerInfo) {
  throw new BadRequestException('CUSTOMER_REQUIRED');
}
```

### 2.3 القيود المحاسبية (Accounting Entries)

القيد الموجود يُغطي هذا السيناريو بالكامل:

```
DR: النقدية (Cash)          50.00 ₪   ← المدفوع نقداً
DR: المدينون (AR)           30.00 ₪   ← المرحَّل آجلاً
CR: إيراد المبيعات (Revenue) 80.00 ₪
```
