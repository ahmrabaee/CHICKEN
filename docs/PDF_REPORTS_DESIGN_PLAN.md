# خطة تحسين تصميم التقارير PDF — Financial Management Program

## الهدف

تحويل التقارير إلى تصميم احترافي عالي الجودة مع دعم كامل لـ RTL وواجهة عربية مميزة، وهيدر غني ببيانات التطبيق.

## المرحلة الأولى: الهيدر (Header) الغني ببيانات التطبيق

### 1.1 هيكل الهيدر الجديد

الهيدر سيحتوي على **ثلاث أقسام أفقية**:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  [شعار]        بيانات التطبيق والشركة              تاريخ التقرير | الصفحة  │
│                                                                             │
│  اسم التطبيق (كبير)     |     اسم المحل/الشركة     |     الرقم الضريبي    │
│  برنامج الإدارة المالية  |     متجر الدجاج          |     3XXXXXXXX       │
│                                                                             │
│  العنوان | الهاتف | البريد | الموقع                  | الفترة: 01-01-2026 |
│  الرياض، شارع الملك فهد   |   +966...  |  info@...  |  إلى 31-01-2026     │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 العناصر المطلوبة في الهيدر

| العنصر | المفتاح (SystemSetting) | ملاحظات |
|--------|-------------------------|---------|
| شعار الشركة | `business_logo` أو `store_logo` | PNG/JPEG، موجود حالياً `assets/logo.png` |
| اسم التطبيق عربي | `app.name` | مثل: برنامج الإدارة المالية |
| اسم التطبيق إنجليزي | `app.name_en` | Financial Management Program |
| إصدار التطبيق | `app.version` | 1.0.0 |

| اسم المحل/الشركة إنجليزي | `business_name_en` | Chicken Shop |
| الرقم الضريبي | `tax_number` أو `tax.registration_number` | |
| العنوان | `business_address` | عنوان النشاط |
| الهاتف | `business_phone` | |
| البريد الإلكتروني | `business_email` | |
| الموقع الإلكتروني | `business_website` | اختياري |
| اسم الفرع | `branchName` (من meta) | |
| تاريخ التوليد | `generatedAt` | |

## المرحلة الثانية: دعم RTL الكامل

### 2.1 المشاكل الحالية (من الكود)

- pdfmake يستخدم `direction: 'rtl'` و `alignment: 'right'` لكن قد لا يكون كافياً
- التعليق يشير إلى: `pdfmake-rtl crashed` — تم الاعتماد على pdfmake العادي
- ترتيب أعمدة الجدول يتم عكسه يدوياً `[...columns].reverse()` عند العربية
- قد يكون هناك خلط بين محاذاة النصوص والأرقام

### 2.2 تحسينات RTL المطلوبة

| العنصر | الإجراء |
|--------|---------|
| اتجاه الصفحة | التأكد من `direction: 'rtl'` على المستوى الافتراضي عندما `language === 'ar'` |
| الهيدر | ترتيب العناصر يمين→يسار: الشعار يمين، التفاصيل وسط، التاريخ يسار |
| الجداول | ترتيب الأعمدة من اليمين لليسار، النصوص محاذاة يمين، الأرقام والعملات محاذاة يسار داخل الخلية (أو وسط) |
| الأقسام المالية | تسميات يمين، القيم يسار أو وسط |
| الـ Footer | نفس منطق الهيدر: شعار يمين، اسم التطبيق وسط، اسم المحل يسار |
| خطوط | التأكد من استخدام خط Cairo (يدعم العربية جيداً) — موجود حالياً ✓ |

### 2.3 حل مشكلة pdfmake-rtl (اختياري)

- تقييم إعادة تفعيل `pdfmake-rtl` إذا كان يعالج الـ bidi بشكل أفضل
- أو الاعتماد على الهيكل اليدوي الحالي مع تحسينات إضافية في الترتيب والمحاذاة

## المرحلة الثالثة: التصميم البصري

### 3.1 نظام ألوان محدث

```typescript
// اقتراح لوحة ألوان احترافية
colors: {
  primary: '#0d47a1',      // أزرق غامق
  secondary: '#1565c0',
  accent: '#1976d2',
  success: '#2e7d32',
  danger: '#c62828',
  warning: '#f9a825',
  text: '#212121',
  textSecondary: '#616161',
  textMuted: '#9e9e9e',
  headerBg: '#e3f2fd',     // أزرق فاتح للجدول
  altRowBg: '#f5f5f5',
  border: '#e0e0e0',
  white: '#ffffff',
}
```

### 3.2 تحسين الجداول

- رأس الجدول: خلفية ملونة (headerBg)، خط عريض، ارتفاع مناسب
- صفوف متناوبة: `altRowBg` للصفوف الزوجية لسهولة القراءة
- حدود: خطوط رفيعة `0.5pt` بين الصفوف، بدون خطوط عمودية (أو خطوط خفيفة)
- Padding: 8–10px لكل خلية
- دعم حالة النص (مثل paid/unpaid): ألوان مختلفة (success/danger) للنصوص

### 3.3 تحسين العناوين والأقسام

- العنوان الرئيسي: حجم 18–20pt، لون primary
- العنوان الفرعي (الفترة): حجم 12pt، لون textSecondary
- عناوين الأقسام المالية: خط تحت (underline) أو شريط جانبي ملون
- تباعد أفضل بين الأقسام

### 3.4 الـ Footer

- شريط سفلي رفيع بلون primary
- صفوف: [شعار | اسم التطبيق + الإصدار | رقم الصفحة]
- حجم خط صغير 7–8pt

### 3.5 Watermark (للإلغاء)

- نص "ملغاة" أو "VOID" شفاف، مائل، بحجم كبير
- لون أحمر خفيف `rgba(200,0,0,0.08)`

## المرحلة الرابعة: الخطوط والطباعة

### 4.1 الخطوط الحالية

- Cairo (عربي + إنجليزي) — ممتاز
- Fallback إلى Arial عند غياب Cairo (ويندوز)

### 4.2 التوصيات

- الحفاظ على Cairo كخط افتراضي
- إضافة خط احتياطي عربي آخر (مثل: Noto Kufi) كخيار في الإعدادات
- أحجام موحدة: title 18, subtitle 12, body 10, footer 7

## المرحلة السادسة: خطة التنفيذ التفصيلية

### الأسبوع 1: الهيدر وبيانات التطبيق

1. **يوم 1–2**: توسيع `PdfMeta` و`getStoreMeta`
   - إضافة حقول: `appName`, `appNameAr`, `appVersion`, `address`, `phone`, `email`, `website`
   - جلب المفاتيح من SystemSetting
2. **يوم 3–4**: إعادة كتابة `buildHeader`
   - هيكل أعمدة: [شعار | بيانات التطبيق + الشركة | التاريخ]
   - دعم RTL: عكس ترتيب الأعمدة عند `language === 'ar'`
   - تجربة على تقرير المبيعات
3. **يوم 5**: مراجعة وضبط الهيدر على مختلف أنواع التقارير

### الأسبوع 2: RTL وتحسين الجداول

1. **يوم 1**: مراجعة اتجاه النصوص في كل المكونات (هيدر، جدول، footer، أقسام مالية)
2. **يوم 2**: تحسين `buildTableSection` — صفوف متناوبة، محاذاة صحيحة للعربية
3. **يوم 3**: تحسين `buildFinancialSections` و`buildSummarySection` لـ RTL
4. **يوم 4**: تحسين `buildFooter` لـ RTL
5. **يوم 5**: اختبار شامل على جميع التقارير

### الأسبوع 3: التصميم البصري

1. **يوم 1**: تحديث `PDF_DESIGN` و`DEFAULT_STYLES`
2. **يوم 2**: تطبيق الألوان الجديدة على الجداول والعناوين
3. **يوم 3**: إضافة تنسيقات للحالات (مدفوع/غير مدفوع) وأي badges
4. **يوم 4**: تحسين الـ Footer والشريط السفلي
5. **يوم 5**: تحسين Watermark وتجربة الطباعة

### الأسبوع 4: الإعدادات والاختبار النهائي

1. **يوم 1–2**: إضافة مفاتيح SystemSetting الجديدة + seed
2. **يوم 3**: واجهة إعدادات (إن وُجدت) أو التأكد من دعم المفاتيح من الـ API
3. **يوم 4–5**: اختبار شامل لجميع أنواع التقارير (15 تقرير)
4. مراجعة الطباعة الفعلية (A4)
