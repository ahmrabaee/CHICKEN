# خطة شاملة لتحسين نقطة البيع `/sales/new`

**التاريخ:** فبراير 2026  
**الصفحة:** http://localhost:1420/sales/new (POS)

---

## 1. تحسين الشكل والترتيب وتوسيع العناصر

### الوضع الحالي
- عمودان: المنتجات (يسار) | السلة (يمين، عرض ثابت 384px)
- عناصر السلة ضيقة
- لا يوجد فصل بصري واضح بين الأقسام

### التحسينات المخططة
| العنصر | التغيير |
|--------|----------|
| **Header** | شريط علوي بعنوان واضح "نقطة البيع" + معلومات الفرع |
| **منطقة المنتجات** | تكبير بطاقات المنتجات، شبكة أكبر، تباعد أوضح |
| **شريط البحث** | توسيع حقل البحث، أيقونات أوضح |
| **الفصل** | استخدام Separator أو حدود أوضح بين: الزبون | السلة | الإجماليات | الأزرار |
| **الأزرار** | تكبير أزرار "نقدي/آجل/مختلط" وطريقة الدفع |
| **الإجماليات** | صفوف أوضح، خط أكبر للإجمالي النهائي |

## 2. توسيع السلة وتكبيرها

### الوضع الحالي
- السلة بعرض `w-96` (384px)
- عناصر السلة: حجم صغير، صفوف مزدحمة

### التحسينات
| العنصر | التغيير |
|--------|----------|
| **عرض السلة** | من `w-96` إلى `min-w-[420px] max-w-[480px]` أو نسبة أكبر (مثلاً 45% من العرض) |
| **ارتفاع السلة** | `min-h` مناسب لملء الشاشة |
| **عناصر السلة** | مساحة أكبر لكل صف، خط أوضح للأسماء والأسعار |
| **منطقة السلة الفارغة** | رسالة واضحة مع أيقونة |

---

## 3. أنصاف الكيلوات (FLOAT للوزن)

### الوضع الحالي
- `quantityKg` رقم صحيح في الكود (addToCart يضيف 1، updateQuantity ±0.5)
- العرض يعرض `item.quantityKg` كعدد
- الـ API يرسل `weightGrams: Math.round(quantityKg * 1000)`

### التحسينات
| العنصر | التغيير |
|--------|----------|
| **CartItem.quantityKg** | الإبقاء كـ `number` يدعم الكسور (2.5، 0.5، إلخ) |
| **إدخال الكمية** | إما حقل رقم مع `step="0.1"` أو `step="0.5"` أو أزرار ±0.5، ±0.25 |
| **التحقق** | السماح بـ `0.1` كحد أدنى (100غرام) |
| **عرض الكمية** | عرض بصيغة `2.5 كجم` أو `2.50 كجم` |
| **الدقة** | استخدام `toFixed(2)` للعرض و `parseFloat` للإدخال |
| **Backend** | `weightGrams: Math.round(quantityKg * 1000)` يبقى صالحاً للكسور |

## 4. البحث عن الزبون

### الوضع الحالي
- Select قائمة منسدلة بجميع الزبائن (100 عنصر)
- لا يوجد بحث داخل القائمة

### التحسينات
| العنصر | التغيير |
|--------|----------|
| **مكون Combobox** | استبدال Select بـ Combobox (بحث + اختيار) |
| **حقل بحث** | إدخال نص للبحث بالاسم/الهاتف |
| **زر بحث** | زر "بحث" بجانب حقل الزبون لفتح قائمة مُفلترة |
| **النتائج** | عرض الزبائن المطابقين مع الاسم والهاتف |
| **Walk-in** | خيار "زبون عادي" يبقى كخيار افتراضي |

## 5. طباعة فاتورة حرارية

### المواصفات
- **العرض:** 58mm (~155–200px) أو 80mm (~302px)
- **الخط:** حجم صغير (12px)، خط بسيط
- **المحتوى:** اسم المحل، تاريخ، بنود، إجماليات، شكراً لكم

### التنفيذ
| المرحلة | الوصف |
|---------|--------|
| **1. مكون ThermalReceipt** | مكون React يعرض الفاتورة بتصميم حراري (عرض ثابت، لا تنسيق معقد) |
| **2. البيانات** | يمكن استخدام بيانات السلة الحالية (قبل الإتمام) أو بعد إنشاء البيع من API `/sales/:id/receipt` |
| **3. window.print()** | `@media print` مع عرض ~200px أو 302px، إخفاء كل شيء ما عدا الفاتورة |
| **4. زر الطباعة** | تفعيل زر "طباعة" بعد إتمام البيع، أو "طباعة معاينة" قبل الإتمام |

### هيكل الفاتورة الحرارية
```
--------------------------------
     [اسم المحل]
     [عنوان / هاتف]
--------------------------------
التاريخ: ...
الزبون: ...
--------------------------------
الصنف        الكم   السعر  الإجمالي
--------------------------------
[بنود...]
--------------------------------
المجموع الفرعي:    ...
الخصم:            ...
الضريبة:          ...
الإجمالي:         ...
المدفوع:          ...
المتبقي:          ...
--------------------------------
     شكراً لكم
     [رسالة تذييل]
--------------------------------
```

## 6. زر "معاينة الفاتورة"

### الوصف
- زر "معاينة الفاتورة" يفتح نافذة/مودال يعرض الفاتورة كما ستُطبع
- يعمل قبل أو بعد إتمام البيع
- قبل الإتمام: يعرض بيانات السلة الحالية
- بعد الإتمام: يعرض بيانات من API إن أمكن

### التنفيذ
| العنصر | التغيير |
|--------|----------|
| **زر** | زر "معاينة الفاتورة" (أيقونة Eye أو FileText) |
| **Dialog/Modal** | نافذة تحتوي على مكون الفاتورة الحرارية |
| **المحتوى** | نفس تصميم الطباعة داخل المودال |
| **خيار طباعة من المعاينة** | زر "طباعة" داخل المعاينة يفعّل `window.print()` على محتوى الفاتورة فقط |
